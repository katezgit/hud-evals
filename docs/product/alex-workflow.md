# Alex's workflow

Five phases Alex moves through when he ships an agent. [`personas.md`](personas.md) tells you who he is; this file tells you what he is doing at each step and what HUD must give him to keep him on the platform. Earlier phases gate later ones — you cannot reward-hack-debug a model whose environment is broken.

> **Conventions used in this doc:**
> - Phases are ordered. Skipping a phase is the failure mode the closing rule names.
> - Each phase has three fields: **Does** (concrete actions in product vocabulary), **Why** (the failure mode this phase prevents or the win it enables), **Surface implication** (what the product surface must do to support this phase).
> - Exactly one phase is the **load-bearing UX of the entire product** for this persona — call it out in its Surface implication. Cuts preserve that phase first.
> - Cite external sources inline when a phase's contract comes from docs outside this repo: `[Source: ...]`.

## Phase 1 — Prompted baseline

- **Does**: writes a V1 of the agent with a prompted frontier model, runs it through `hud eval` against the eval set, manually inspects a sample of rollouts. [Source: `hudevals/docs/product/aman-workflow.md`; `hud eval tasks.py` command confirmed at `https://docs.hud.ai/building/tasks-and-evaluation.md`]
- **Why**: separates environment bugs from policy weakness. If the prompted baseline clears the bar, RL never starts and he ships faster. If it doesn't, the gap tells him how much RL needs to lift, and any tools that aren't returning the right data surface now — not 8 hours into training.
- **Surface implication**: `hud eval` is a first-class entry point, not a precursor to `hud rl`. The web app needs a per-rollout view from day one (tool calls, arguments, returned payloads, prompt + response) so he can confirm tools are wired correctly before committing to training. [Source: quickstart onboarding at `https://docs.hud.ai/llm-quickstart.md` confirms `hud eval` as the first substantive product action after `hud init`]

## Phase 2 — Realistic environment + Tool authoring

- **Does**: ingests messy, production-shape data (public corpora like the Enron emails, scraped logs, real-shape fixtures) and wires it into the env definition. Simultaneously authors the Tools the agent will call against that data — `@env.tool()` decorators whose docstrings + type hints become the agent-facing signature, `@tool.before` / `@tool.after` hooks for validation and logging, `BaseTool` subclasses for stateful interactions. Data and Tools are co-authored: the data shapes what the Tools operate on, and the Tool signatures constrain what data shapes are valid. [Source: `hudevals/docs/product/aman-workflow.md`; "Environments as Data" concept at `https://docs.hud.ai/building/environments-as-data.md`; Tool authoring via `@env.tool()` decorator confirmed at `https://docs.hud.ai/building/scaffolding`]
- **Why**: two compounding failure modes live here. (1) An agent trained on synthetic-looking data optimizes for the wrong distribution and fails in prod — he would rather debug a messy real dataset than a clean fake one. (2) If the Tool surface differs from production — wrong signatures, missing hooks, stateful interactions that aren't modeled — the agent learns to use Tools that won't exist at deploy time. A bug in the Tool definition is a silent bug in every subsequent Run: the agent appears to learn, but it is learning the wrong thing against a tool that isn't real.
- **Surface implication**: env authoring affordances need to support Tool authoring as a first-class activity alongside data ingestion — not CLI-only. Preview and inspect the Tools the agent has access to; validate signatures before committing to training; surface `@tool.before` / `@tool.after` hook failures inline, not in a log file. The Marketplace must preserve Tool wiring when forking an existing Environment, not just data wiring — a fork that resets the Tools is silently broken.

## Phase 3 — Scenario authoring + grader validation

- **Does**: authors the Scenarios that define what counts as success — the two-yield contract where yield 1 sends the prompt to the agent and yield 2 inspects environment state and returns a reward (`1.0 if order_exists else 0.0`). He composes Graders (programmatic, state-based checks: computer state, filesystem, coding, web) to express the reward logic. He then validates that the Grader actually fires correctly on known cases before scaling to a full Taskset. [Source: `hudevals/docs/product/aman-workflow.md`; two-yield pattern confirmed at `https://docs.hud.ai/building/tasks-and-evaluation.md`; Native Graders reference at `https://docs.hud.ai/reference/native-graders.md`] When the Environment data is unstructured (emails, scraped web pages, real corpora), he also bootstraps a golden Task set using LLM-assisted generation: he feeds batches of raw data samples to a frontier model (e.g. 20 Enron emails per call to Gemini 2.5 Pro) and asks it to produce realistic user questions with verified answers, then filters out unrealistic generations and commits the curated set to the Taskset. This is **LLM-assisted Task generation, human-curated** — not LLM-judge-at-eval-time (which is `LLMJudgeGrader`'s job). [Source: Corbitt, "How to Train Your Agent" (AI Engineer Conference 2025): "fed batches of 20 Enron emails to Gemini 2.5 Pro and asked it to generate realistic user questions along with their verified answers. After filtering out the weird questions, he was left with a golden dataset of a few thousand verified queries" — ART·E email research agent, OpenPipe]
- **Why**: the Scenario + Grader pair is ground truth for every subsequent Run. If the reward function is wrong — it fires on the wrong condition, or never fires at all — every training run is wrong. Grader validation before scaling is the equivalent of unit-testing the test harness. HUD's native Grader set includes both code-defined deterministic Graders (`BashGrader`, `exact_match`, `contains`, `numeric_match`, `f1_score`) and `LLMJudgeGrader` (rubric-based, LLM-evaluated) — all return a numerical float. Alex's profile (verifiability above all) tends toward code-defined Graders where the success condition can be expressed in code; `LLMJudgeGrader` is the right tool when only a rubric is available. This is the design implication for the Phase 3 surface: default affordances should make code-defined Grader composition easy and auditable. [Source: docs.hud.ai/building/scaffolding — Native Graders canonical set] [Reconciliation: hudevals `personas.md` uses "golden data + LLM judge" as a shorthand; `LLMJudgeGrader` is a real native primitive — the correction is that it coexists with code-defined Graders, not that it is absent]
- **Surface implication**: HUD needs a surface for authoring, previewing, and validating Scenario + Grader pairs on the Environment — not a CLI-only workflow. Alex needs to see: the Scenario definition, the Grader logic, and a test harness that lets him run the Grader against a sample of real environment states and inspect the reward signal. A "Grader fired / didn't fire" diff view before full-scale eval is the load-bearing piece of this phase. Scenarios are decorated on the Environment (`@env.scenario`); Tasks (Scenarios instantiated with specific argument values) are what get collected into a Taskset. The Taskset detail page also needs a "Generate Tasks from Environment data" affordance: sample N rows of Environment data, run an LLM to propose Tasks (question + expected answer), and present the candidates in a review queue where Alex can accept, reject, or edit each one before committing to the Taskset. This is distinct from the Grader validation panel — generation happens before validation, and the human curation step is the gate that catches unrealistic candidates before they poison the eval set.

## Phase 4 — Training loop + multi-objective reward

- **Does**: runs `hud rl`, watches loss / reward curves, layers in secondary rewards — fewer tool turns (latency, cost), a **honesty penalty** that rewards the agent for saying "I don't know" over fabricating an answer when its tools didn't surface ground truth, tighter answer length — and adjusts weights between runs. [Source: `hudevals/docs/product/aman-workflow.md`; Corbitt, "How to Train Your Agent" (AI Engineer Conference 2025): honesty penalty as a named loss term in ART·E's reward function — rewarding "I don't know" over fabrication when retrieval failed]
- **Why**: accuracy alone is not the win condition. He is cutting the frontier-model bill down by 10x; latency and tool turns are part of the cost calculation. Honesty penalties keep the policy from hallucinating when retrieval comes up empty. These secondary rewards are not nice-to-haves: ART·E achieved higher accuracy, lower cost, and lower latency than OpenAI o3 on email search specifically because of joint optimization across all objectives, not despite it. [Source: Corbitt, "How to Train Your Agent" (AI Engineer Conference 2025) — ART·E benchmark results vs o3]
- **Surface implication**: training-curve dashboard must surface more than reward. Cost-per-rollout, average latency, tool-turn count, and hallucination rate are first-class series alongside reward — each with a one-click drill to the rollouts driving it. Reward weights are visible and editable next to the curve, not buried in a config file.

## Phase 5 — Reward-hack inspection

- **Does**: watches for unnatural curve shapes — a sudden spike to near-perfect score, a suspicious flatline, a metric climbing too fast — and clicks into the rollouts behind that point on the graph to read what the policy is actually doing. [Source: `hudevals/docs/product/aman-workflow.md`; reward hacking listed as a named analysis query in Trace Analysis at `https://docs.hud.ai/platform/internal/trace-analysis.md`: "Is there evidence of reward hacking or gaming the evaluation?"] When a hack is identified, the standard fix is to **edit the Grader** — add a sub-Grader that penalizes the discovered loophole (e.g. an LLM judge that checks whether a generated title is actually supported by the article content). This loops Alex back to Phase 3 (Grader validation against the new sub-Grader) and then re-launches training. Phase 5 is not terminal; the discovered-hack → patched-Grader → re-train edge is a load-bearing loop in the workflow. [Source: Corbitt, "How to Train Your Agent" (AI Engineer Conference 2025): "When you catch the model reward hacking, the standard debugging fix is to modify your reward function. For the Hacker News title generator, Corbitt fixed the exploit by adding a secondary LLM judge specifically instructed to penalize the model if the title was not supported by the underlying article content"]
- **Why**: this is where models cheat. Two distinct hack shapes appear in practice: (a) **constant-output exploitation** — a policy learns to output "Google lays off 80% of workforce" for every input because it games a noisy judge; (b) **maximal-coverage exploitation** — a categorization policy places every word into every category to maximize a coverage-based partial-credit reward. Curves alone will not catch either; only the actual generations will. [Source: Corbitt, "How to Train Your Agent" (AI Engineer Conference 2025) — both examples named explicitly]
- **Surface implication**: **this is the load-bearing UX of the entire product for Alex.** From any data point on any training curve, one click opens the rollouts (Traces) at that step — tool calls, arguments, returned payloads, prompts, responses, and the reward signal from the Grader. No modals layered three deep. No re-querying. No "open in another tab." A reward-hack inspection that takes more than one click is the reason he writes his own harness and tells his team HUD is not the answer. [Source: Trace Analysis surface confirmed at `https://docs.hud.ai/platform/internal/trace-analysis.md` — "tool calls, errors, agent turns" are named trace-level fields; Analysis Tab provides per-trace custom queries] The surface must also expose a path from a flagged Trace back to the Grader editor (Phase 3 surface) — a contextual affordance on the Trace detail view that opens the Grader source for the Grader that fired on that Run, with the flagged Run preloaded as a validation sample. This is not a "patch reward function" button; it is a contextual link: "this hack is happening; here is the Grader configuration that allowed it; edit and revalidate."

---

*Designing flows that skip a phase — e.g., a "start training" wizard that doesn't run through `hud eval` first — breaks the workflow Alex trusts. Phase order is the workflow.*

*Phases are ordered but not strictly linear: Phase 5 → Phase 3 → Phase 4 is the standard reward-hacking-debugging loop. The ordering constraint governs entry (you cannot start at Phase 4 with a broken Phase 2 environment), not re-entry.*
